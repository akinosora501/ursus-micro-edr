auth Auth Collector

systemd-journald を journalctl 経由で購読し SSH/sudo を観測

目的

認証イベントは EDR の中でも特に重要です。攻撃者の足跡として:

  • SSH ブルートフォース: 短時間に大量の Failed password
  • 初期アクセス確立: Accepted password を観測
  • 権限昇格: sudo による特権コマンド実行
  • 不審なログイン: 通常使わないアカウントや時刻のログイン

採用方式: journalctl 購読

DESIGN.md の初版では /var/log/auth.log を tail する想定でしたが、最近の ディストリ（Arch, Fedora, Parrot OS 等）では auth.log を持たず、 systemd-journald に統合されています。URSUS は journalctl -f --output=json を子プロセスとして起動し、その stdout から JSON を 1 行ずつ受け取ります。

方式環境依存追加依存備考
journalctl 購読 (採用) systemd 必須 なし（journalctl は同梱） auth.log が無くても動く
auth.log tail rsyslog 必要 なし 古い実装。最近のディストリで動かない
libsystemd の python binding systemd 必須 python-systemd API クリーンだが C 拡張のビルドが必要
NETLINK_AUDIT auditd 必要 なし システム auditd と排他

なぜ subprocess + select?

journalctl -f は内部で sd_journal_wait() を呼び、 カーネルの inotify / signalfd 通知で叩き起こされる形で寝ています。 これにより新規ログが書かれた瞬間に stdout に流せる効率実装になっています。

URSUS から見ると:

  • journalctl 自体は寝ていて CPU を使わない
  • 新規行は数 ms 以内に届く（取りこぼし・遅延ともなし）
  • Python 側は select.poll() で stdout fd を待ち、500ms ごとに stop_event を確認する
   journalctl -f --output=json     ─────  カーネル
   ────────────────────────────              │
        │                                    │ inotify on /var/log/journal/
        │  sd_journal_wait()                 │ + signalfd
        │      │                             │
        │      └────── 寝る ─────── 起こされる ◀┘
        ▼
   stdout (1行 1JSON)
        │
        ▼
   Python: select.poll(0.5s)  ─────── stop_event チェック
        │
        ▼
   _handle_line(line): JSON.loads → MESSAGE 抽出 → 正規表現で分類
        │
        ▼
   events 表に INSERT

journalctl の match 構文で絞り込む

journalctl は FIELD=VALUE 形式で match を指定すると kernel 側で フィルタリングしてくれます。URSUS は config の journal_units / journal_comms から自動生成します。 

# 起動するコマンド (実際には stdbuf + journalctl)
journalctl -f --output=json --no-pager \
    _SYSTEMD_UNIT=ssh.service \
    _SYSTEMD_UNIT=sshd.service \
    + \
    _COMM=sudo
journalctl の match セマンティクス
  • 同じフィールドの match: OR 結合 (例: _SYSTEMD_UNIT=A _SYSTEMD_UNIT=B = A or B)
  • 異なるフィールドの match: AND 結合
  • + を境にすると左右のグループ全体が OR で結ばれる
  • + は FIELD=VALUE 同士の間でしか使えない (-u オプションは隣に置けない)

ログ行のパース

journalctl が JSON で返してくれる MESSAGE フィールドを正規表現で 分類します。URSUS は 3 パターンを認識します。 

RE_ACCEPTED = re.compile(r'\bAccepted\s+(\S+)\s+for\s+(\S+)\s+from\s+(\S+)')
RE_FAILED   = re.compile(r'\bFailed\s+(\S+)\s+for\s+(?:invalid user\s+)?(\S+)\s+from\s+(\S+)')
RE_SUDO     = re.compile(r'^\s*(\S+)\s+:\s+.*COMMAND=(.+)$')
パターンマッチ例結果
Accepted Accepted password for root from 1.2.3.4 port 22 ssh2 service=sshd, result=success, user=root, source_ip=1.2.3.4
Failed Failed password for invalid user admin from 1.2.3.4 port 22 ssh2 service=sshd, result=failure, user=admin, source_ip=1.2.3.4
sudo parrot : TTY=pts/0 ; PWD=/home ; USER=root ; COMMAND=/usr/bin/cat /etc/shadow service=sudo, result=sudo, user=parrot, command=/usr/bin/cat /etc/shadow

sudo を判別する hint

SSH の Accepted / Failed はキーワードで自明に見分けられますが、 sudo の MESSAGE は parrot : TTY=... のようにユーザー名で始まり、 プレフィックスから "sudo のログ" だと判別できません。journalctl が一緒に 返す _COMM フィールド（プロセス名）が sudo のときだけ sudo ルールで パースする、という hint 方式を取っています。 

def parse_line(line, service_hint=None):
    if service_hint == "sudo":
        m = RE_SUDO.search(line)
        ...

URSUS の実装

ファイル: src/ursus/sensor/journal_auth_collector.py 

def _read_loop(self, conn, proc, stop_event):
    fd = proc.stdout.fileno()
    poller = select.poll()
    poller.register(fd, select.POLLIN | select.POLLHUP | select.POLLERR)

    while not stop_event.is_set():
        events = poller.poll(500)  # 500ms は stop_event 確認用
        if not events:
            if proc.poll() is not None:
                return  # journalctl が早期終了
            continue
        for _fd, mask in events:
            if mask & select.POLLIN:
                line = proc.stdout.readline()
                self._handle_line(conn, line)

設定

sensor:
  auth:
    enabled: true
    journal_units:
      - ssh.service
      - sshd.service        # Debian/Ubuntu と RHEL の両方をカバー
    journal_comms:
      - sudo

他のサービス（例: PAM, polkit, GDM のログイン）も追加で取りたい場合は journal_units / journal_comms を増やせます。

制限

  • journalctl が stdbuf 経由でないとブロックバッファリングする: URSUS は stdbuf -oL 付きで起動。これにより stdout が行バッファ化され、 1 行ずつリアルタイムに届く。
  • 非 UTF-8 のログ: journald は MESSAGE をバイト配列で返すことがある。 URSUS は bytes.decode("utf-8", "replace") で救済。
  • 正規表現でマッチしないログは捨てる: 知らないフォーマットの認証ログは 記録されない。新しいパターンに遭遇したら parse_line に追加する設計。
  • R006 (時間窓型ルール): 「短時間に N 回失敗」のような閾値型検知は 現在の DSL では表現できない。condition に time_window を追加する拡張で対応可能。

拡張余地

  • PAM event 拡張: ログイン以外 (パスワード変更、アカウント追加) も拾う
  • libsystemd 直接バインド: subprocess を介さず Python から sd_journal API を直接呼ぶ
  • auditd との統合: より細かいシステムコール監視
  • 時間窓型ルール: ブルートフォースなどの集計型検知
