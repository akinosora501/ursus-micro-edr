network Network Collector

psutil による TCP 接続テーブル差分ポーリング

目的

攻撃者の通信は以下の典型パターンで観測できます:

  • 新規 LISTEN: 攻撃者がリバースシェル待ち受け、または不要なポートを開いた
  • 新規 ESTABLISHED (outbound): マルウェアの C2 通信、データ送信
  • 新規 ESTABLISHED (inbound): 横展開・攻撃の侵入経路

URSUS は psutil.net_connections(kind='tcp') で TCP 接続テーブルの スナップショットを取り、前回との差分から「新規」を抽出します。

採用方式: psutil 差分ポーリング

他のセンサが kernel push 通知を使うのに対し、network だけは ポーリングです。これは Linux に「TCP 接続生成の汎用イベント ブロードキャスト」が用意されていない歴史的事情によります。

方式取りこぼしPID 取得備考
psutil ポーリング (採用) あり ◎ 2 秒間隔で /proc/net/tcp 系を読む
NETLINK_SOCK_DIAG 同上 ◎ ss(8) が使う API。生成イベントは取れない
NETLINK_NETFILTER (conntrack) なし（イベント駆動） ✗ conntrack は kernel 側のエントリを通知。PID は取れない
NETLINK_AUDIT なし ◎ auditctl -S connect -S accept。auditd と排他
eBPF (kprobe) なし ◎ + 親プロセス tcp_v4_connect, inet_csk_accept。商用 EDR 標準
取りこぼしを許容している

ポーリング間隔（既定 2 秒）より短命な接続は検知できません。curl http://... が 1 秒以内に閉じる接続は見えないことがあります。 差分ポーリング方式に固有の制限。eBPF (kprobe tcp_v4_connect / inet_csk_accept) に置き換えれば解消する。

差分検出のしくみ

   poll cycle N        poll cycle N+1
   ─────────────       ──────────────
   {A, B, C, D}        {A, B, C, E, F}     ← 現在のスナップショット

                        new = current - known
                        new = {E, F}        ← イベント化対象

   _known = {A, B, C, D}  →  _known = {A, B, C, E, F}

起動時の既存接続をベースラインとし、以降のポーリング差分のみを記録

接続を識別するキー

ある接続が「新規かどうか」を判定するには、識別キーが必要です。URSUS は 4-tuple + status + pid を使います。 

def _conn_key(sc):
    laddr = (sc.laddr.ip, sc.laddr.port) if sc.laddr else None
    raddr = (sc.raddr.ip, sc.raddr.port) if sc.raddr else None
    return (laddr, raddr, sc.status, sc.pid)

status を含めることで、同じ接続が SYN_SENT → ESTABLISHED と 状態遷移したときに「新しい ESTABLISHED」として正しく検出されます。

記録対象の状態

TCP の状態は 11 種類ありますが、URSUS は LISTEN と ESTABLISHED のみを 記録対象にしています。

状態意味記録
LISTENサーバが接続待ち◯
SYN_SENTクライアントが接続要求送信中✗ ESTABLISHED に進んだら拾う
SYN_RECVサーバが SYN 受信、SYN+ACK 返した✗ 同上
ESTABLISHED接続確立中（双方向通信可能）◯
TIME_WAITクローズ後の待機✗ 高頻度ノイズ
CLOSE_WAIT ほか遷移状態✗

ESTABLISHED の重複抑制

HTTP/2 connection pool やメッセンジャーアプリの再接続では、同じプロセスが 同じ宛先 IP:Port に対して短時間に多数の接続を開閉します。ephemeral port が 毎回違うため 4-tuple は全て異なりますが、検知の観点ではノイズです。

URSUS は (pid, remote_ip, remote_port) をキーに、 established_debounce_sec 窓内の重複を破棄します。LISTEN は 低頻度かつ高シグナルなので debounce 対象外です。 

def _should_emit(self, sc):
    if sc.status != "ESTABLISHED": return True
    if self.established_debounce_sec <= 0: return True
    if not sc.raddr: return True
    key = (sc.pid, sc.raddr.ip, sc.raddr.port)
    now = time.monotonic()
    last = self._last_established_emit.get(key)
    if last is not None and now - last < self.established_debounce_sec:
        return False
    self._last_established_emit[key] = now
    return True

URSUS の実装

ファイル: src/ursus/sensor/network_collector.py 

def run(self, stop_event):
    conn = get_connection(self.db_path)
    # 起動時の既存接続はベースラインとして黙って取り込む
    self._known = self._snapshot_keys()

    while not stop_event.is_set():
        if stop_event.wait(self.poll_interval):
            break

        snapshot = psutil.net_connections(kind="tcp")
        current = set()
        new_conns = []
        for sc in snapshot:
            if sc.status not in _TRACKED_STATES:
                continue
            key = _conn_key(sc)
            current.add(key)
            if key not in self._known:
                new_conns.append(sc)

        for sc in new_conns:
            if not self._should_emit(sc):  # debounce
                continue
            self._record(conn, sc)

        self._known = current
        self._gc_debounce()

LISTEN は raddr が NULL

LISTEN ソケットは「待ち受けているだけで誰とも繋がっていない」ので remote_addr / remote_port が NULL になります。UI の Detail 列は LISTEN だったら laddr を表示するように分岐しています。

設定

sensor:
  network:
    enabled: true
    poll_interval_sec: 2.0
    established_debounce_sec: 5.0   # 0 で無効化

制限

  • UDP は対象外: DNS (53) / NTP (123) / QUIC は見えない
  • ICMP も対象外: ping / traceroute は記録されない
  • ペイロード非観測: 4-tuple のみ。HTTP リクエスト本体や TLS SNI は見えない
  • 取りこぼし: poll 間隔より短命な接続

DNS を観測したい場合

URSUS の現在の実装では DNS クエリは取れません。観測する場合の選択肢:

  1. パケットキャプチャ: scapy / pyshark で UDP/53 を直接観察
  2. systemd-resolved のログ: journalctl -u systemd-resolved で DNS 解決ログ。auth_collector と同じ journalctl パターンで実装可能
  3. eBPF: udp_sendmsg kprobe で UDP/53 をフック
  4. DNS resolver ホスト化: dnsmasq などを挟んでログ取得

商用 EDR は概ね eBPF + パケットキャプチャの併用です。
