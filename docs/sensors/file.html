file File Collector

inotify (watchdog) によるファイル変更のリアルタイム検知

目的

攻撃者の永続化やデータ改竄は、最終的に「ファイル書き込み」として表面化します。 代表的な観測対象:

  • /etc/passwd や /etc/shadow への書き込み (バックドアアカウント追加)
  • ~/.ssh/authorized_keys の改変 (SSH 鍵による持続的アクセス)
  • /etc/cron.d/, /etc/systemd/system/ への投入 (定期実行による永続化)
  • /tmp や /var/tmp 配下への payload 配置
  • ~/.bash_history や /var/log/* の消去 (痕跡隠蔽)

採用方式: inotify (watchdog 経由)

Linux でファイル変更を観測する選択肢:

方式PID/Process 取得必要権限備考
inotify (採用) ✗ 不要 カーネルの fs 通知。watchdog ライブラリで Python から扱う
fanotify ◎ (PID/exe path) CAP_SYS_ADMIN 5.1+ で FAN_REPORT_FID。Python 標準には無く ctypes 必要
auditd file watch ◎ root auditctl -w /etc/passwd -p wa で監視ルール設定
eBPF (LSM hook) ◎ + コンテナ情報 CAP_BPF 商用 EDR の主流実装。bcc / libbpf 等への依存が必要
inotify は "誰が変更したか" を返さない

URSUS の現在の実装では、/etc/passwd が改変されたことは分かりますが 「どのプロセスが書き込んだか」は分かりません。inotify 自体がその情報を 返さないからです。fanotify / auditd / eBPF への発展はこの制約を解消する ためのものです。

inotify とは

inotify は Linux 2.6.13 から導入されたファイルシステム 通知機構です。inotify_init() で得たファイルディスクリプタに対して inotify_add_watch() で監視対象パスを登録すると、その配下で ファイルが作成・変更・削除・移動された際にイベントが配信されます。

イベント種別 (mask):

mask発火タイミングURSUS の扱い
IN_CREATEファイル/dir 作成create として記録
IN_MODIFYwrite(2) ごと（!）modify として記録（debounce あり）
IN_CLOSE_WRITE書き込み用 fd の close未使用
IN_DELETE削除delete として記録
IN_MOVED_FROM/TOrename(2)move として記録 (src + dest)
IN_ATTRIBchmod/chown未使用（SUID 検知は process 経由で行う）

watchdog ライブラリの位置付け

URSUS は inotify を直接叩くのではなく、Python の watchdog ライブラリを使っています。watchdog は OS 抽象化レイヤを提供し、 Linux では InotifyEmitter、macOS では FSEvents、Windows では ReadDirectoryChangesW を使います。我々が必要なのは Linux 限定なので inotify 直叩きでも構わないのですが、コード量を減らすために採用しました。

スレッドモデルの注意点

   watchdog Observer (別スレッド)         FileCollector.run() (sensor スレッド)
   ─────────────────────────              ─────────────────────────────────────

   inotify 受信
        │
        ▼
   _FileEventHandler.on_modified()        out_queue.get(timeout=0.5)
        │                                          │
        │  out_queue.put(payload)                  │
        ├─────────────────────────────────────────▶│
        ▼                                          ▼
   (続けて受信ループ)                        DB INSERT (sqlite3.Connection)

Observer 側スレッドと DB 書き込みスレッドを Queue で疎結合にする

watchdog の FileSystemEventHandler のコールバックは Observer の 別スレッドで呼ばれます。一方 SQLite の接続オブジェクトはスレッドをまたいで 共有しないのが安全な使い方です。URSUS では queue.Queue を間に 挟むことで:

  • Observer スレッドはイベントを Queue に積むだけ
  • run() スレッドが Queue から取り出して DB に INSERT

これによりロックの取り合いを避け、終了処理（SIGTERM）のときも main スレッドの ループ条件で素直に止められます。

URSUS の実装

ファイル: src/ursus/sensor/file_collector.py

イベントハンドラ

class _FileEventHandler(FileSystemEventHandler):
    def on_created(self, event):
        self._emit("create", event.src_path, event.is_directory)

    def on_modified(self, event):
        # ディレクトリの modified は配下のファイル変更ごとに発火し
        # ノイズが多いため捨てる。
        if event.is_directory: return
        if self._is_excluded(event.src_path): return
        if not self._should_emit_modify(event.src_path): return
        self._queue.put({"op": "modify", "path": event.src_path, ...})

    def on_deleted(self, event): ...
    def on_moved(self, event):   ...

除外パターンと自己フィードバック防止

何も対策しないと URSUS 自体が data/edr.db-wal を秒間数千回更新する ので、ファイル監視がそれを拾って無限ループになります。コードでは:

  1. DB ディレクトリは無条件除外: os.path.realpath で正規化して、watch_path 配下に DB がある 場合でも自動的にスキップ。SQLite の WAL/SHM/journal が同一ディレクトリに 作られるので、ディレクトリ単位で切れば全部カバーできる。
  2. glob パターン除外: config.yml の exclude_patterns に指定された glob （例: *.swp, *~, */.cache/*）に マッチするパスを除外。
def _is_excluded(self, path):
    try:
        real = os.path.realpath(path)
    except OSError:
        real = path
    for d in self._exclude_dirs:
        if real == d or real.startswith(d + os.sep):
            return True
    name = os.path.basename(path)
    for pattern in self._exclude_patterns:
        if fnmatch.fnmatch(name, pattern) or fnmatch.fnmatch(path, pattern):
            return True
    return False

modify の重複抑制 (debounce)

inotify の IN_MODIFY はファイルへの write(2) 1 回ごとに 発火します。エディタが 1 ファイル保存しただけで数十イベント出ることが 珍しくありません。URSUS は同一 path への連続 modify を modify_debounce_sec 窓内で 1 件にまとめます。 

def _should_emit_modify(self, path):
    if self._debounce_sec <= 0:
        return True
    now = time.monotonic()
    with self._lock:
        last = self._last_modify_emit.get(path)
        if last is not None and now - last < self._debounce_sec:
            return False
        self._last_modify_emit[path] = now
    return True
商用 EDR の "file write storm suppression"

商用 EDR は同様の抑制を「ウィンドウ末で破棄件数 N を 1 件追加で出す」形で 実装することが多い (leading + trailing summary)。URSUS は実装の単純さを 優先して leading-only に留めている。トレーリング合成への拡張は容易。

出力されるイベント

raw_json

{
  "op": "modify",
  "path": "/etc/passwd",
  "is_directory": false,
  "src_path": null
}

op は create / modify / delete / move。 move 時のみ src_path に移動元、path に移動先が入ります。

非正規化カラム

カラム例
file_path/etc/passwd
file_opcreate / modify / delete / move

設定

sensor:
  file:
    enabled: true
    watch_paths:                 # 再帰監視するディレクトリ
      - /etc
      - /tmp
      - /var/tmp
      - /home
    exclude_patterns:            # glob 除外
      - "*.swp"
      - "*~"
      - "*/.cache/*"
      - "*/.mozilla/*"
    modify_debounce_sec: 2.0     # 同一 path への連続 modify を抑制

運用上のチューニング

watch_paths を絞る

/home 全体を監視するとブラウザキャッシュ・エディタ作業ファイル等で 数千イベント/秒が出ることがあります。攻撃者の永続化候補を狙い撃ちすると 精度とノイズのバランスが取れます。 

watch_paths:
  - /etc
  - /etc/cron.d
  - /etc/cron.daily
  - /etc/systemd/system
  - /usr/bin
  - /usr/sbin
  - /tmp
  - /var/tmp
  # /home 全体ではなく重要パスだけピン留め
  - /home/user/.ssh
  - /home/user/.bashrc
  - /home/user/.profile

exclude_patterns を厚く

ノイズの多い拡張子・ディレクトリは exclude で叩き落とします: 

exclude_patterns:
  - "*.swp"
  - "*~"
  - "*.lock"
  - "*.pyc"
  - "*/.cache/*"
  - "*/.mozilla/*"
  - "*/.git/*"
  - "*/node_modules/*"
  - "*/__pycache__/*"

制限

  • PID が分からない: 上述。fanotify / auditd / eBPF へ拡張する余地
  • recursive watch のコスト: 再帰監視は kernel 側で各サブディレクトリごとに watch descriptor を 張ります。/home に膨大な dir があると fs.inotify.max_user_watches の上限（デフォルト 8192）に達することがある。
  • NFS / FUSE の挙動差: 一部のネットワークファイルシステムでは inotify が完全には動きません
