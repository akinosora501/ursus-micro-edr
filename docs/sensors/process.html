process Process Collector

netlink (cn_proc) でプロセス起動をカーネルから push 通知で受け取る

目的

URSUS が観測する 4 つのイベントタイプの中で最も重要なのが プロセス起動です。攻撃者の行動はほぼ必ず「何かのプロセスを起動する」という 形で表面化します。シェル、payload バイナリ、永続化のための cron タスク、 横展開の psexec — 全てプロセス起動として観測できます。

採用方式: cn_proc netlink

プロセス起動を観測する Linux 上の選択肢は複数あります。URSUS は kernel から直接 push 通知を受ける方式を採用しています。

方式取りこぼし必要権限備考
cn_proc netlink (採用) なし CAP_NET_ADMIN カーネルが exec(2) のたびに通知。リアルタイム
psutil ポーリング あり (poll 間隔より短命) 不要 DESIGN.md の初版方針。実装簡単だが取りこぼし大
auditd なし root execve syscall を audit ルールで購読。出力フォーマットが煩雑
eBPF (kprobe) なし CAP_BPF 商用 EDR の主流実装。bcc / libbpf-CO-RE への依存が必要

cn_proc とは

cn_proc (process events connector) は、Linux カーネルが connector インタフェースを介して送出するマルチキャストイベントです。netlink ソケットの NETLINK_CONNECTOR プロトコルファミリーで購読します。

カーネルは以下のイベントを通知します:

  • PROC_EVENT_FORK: fork(2) 完了時
  • PROC_EVENT_EXEC: execve(2) 成功時
  • PROC_EVENT_EXIT: プロセス終了時
  • PROC_EVENT_UID, PROC_EVENT_GID: 権限変更時
  • その他 (sid, ptrace 等)

URSUS は EXEC のみを購読しています。検知に必要な情報 （プロセス名・コマンドライン・実行ファイル・親プロセス）は exec の タイミングで揃うためです。fork だけでは bash のプロセスが何回も fork した中身が分かりません。

netlink メッセージの構造

netlink で受け取るバイト列は 3 段の入れ子になっています。Python 標準 ライブラリの struct モジュールで unpack して読み取ります。 

  bytes 0  ─┐
            │  ┌──────────────────────────────┐
   nlmsghdr │  │ len, type, flags, seq, pid   │  16 bytes
            │  └──────────────────────────────┘
  bytes 16 ─┤
            │  ┌──────────────────────────────┐
   cn_msg   │  │ idx, val, seq, ack, len, fl  │  20 bytes
            │  └──────────────────────────────┘
  bytes 36 ─┤
            │  ┌──────────────────────────────┐
   evt_hdr  │  │ what, cpu, timestamp_ns      │  16 bytes
            │  └──────────────────────────────┘
  bytes 52 ─┤
            │  ┌──────────────────────────────┐
  body (exec)│ │ pid, tgid                    │  8 bytes
            │  └──────────────────────────────┘
  bytes 60 ─┘

PROC_EVENT_EXEC を含む netlink メッセージのレイアウト

pid と tgid

exec イベントの body には pid と tgid の 2 つが入っています。 Linux ではスレッドも軽量プロセスとして扱われるため、スレッドからの exec も通知が来ます。我々が「プロセス」と呼んでいる単位は tgid （thread group id、ユーザー空間からは getpid() で見える PID）に対応します。 スレッド単位では tgid が同じで pid だけ異なるイベントが上がるため、 取りこぼしを避けつつ重複も避けるには tgid で集約します。

URSUS の実装

ファイル: src/ursus/sensor/process_collector_netlink.py

手順 1: netlink ソケットを開いて bind

def _open_socket(self):
    s = socket.socket(socket.AF_NETLINK, socket.SOCK_DGRAM, NETLINK_CONNECTOR)
    # bind: pid=0 でカーネル自動採番、groups=CN_IDX_PROC で connector の
    # プロセスイベントマルチキャストを購読。
    s.bind((0, CN_IDX_PROC))
    return s

通常の TCP/UDP ソケットと同じ socket.socket ですが、 AF_NETLINK ファミリーを指定し、プロトコル番号に NETLINK_CONNECTOR = 11 を渡します。bind の groups 引数は 購読するマルチキャストグループのビットマスクです。

手順 2: 「購読開始」を kernel に通知

netlink ソケットを開いて bind しただけでは、まだイベントは流れて 来ません。明示的に PROC_CN_MCAST_LISTEN を kernel に送ることで 配信が始まります。 

def _send_op(self, op):
    body = struct.pack("=I", op)
    cn_payload = _CNMSG.pack(
        CN_IDX_PROC, CN_VAL_PROC,    # cb_id
        0, 0,                        # seq, ack
        len(body), 0,                # len, flags
    ) + body
    total_len = _NLMSGHDR.size + len(cn_payload)
    nl_hdr = _NLMSGHDR.pack(total_len, NLMSG_DONE, 0, 0, 0)
    self._sock.send(nl_hdr + cn_payload)

手順 3: 受信ループ

self._sock.settimeout(0.2)  # stop_event を 200ms ごとに見るため
while not stop_event.is_set():
    try:
        data, _ = self._sock.recvfrom(8192)
    except socket.timeout:
        continue
    self._handle_message(data)

手順 4: バイト列を unpack

def _handle_message(self, data):
    # nlmsghdr はスキップ。cn_msg ヘッダから idx/val を確認。
    idx, val, _seq, _ack, _payload_len, _flags = _CNMSG.unpack_from(
        data, _NLMSGHDR.size
    )
    if idx != CN_IDX_PROC or val != CN_VAL_PROC:
        return

    evt_off = _NLMSGHDR.size + _CNMSG.size
    what, _cpu, _ts_ns = _EVTHDR.unpack_from(data, evt_off)
    body_off = evt_off + _EVTHDR.size

    if what == PROC_EVENT_EXEC:
        pid, tgid = struct.unpack_from("=II", data, body_off)
        self._record_exec(tgid)

手順 5: /proc から詳細情報を取得

netlink から来るのは pid だけです。コマンドラインや実行ファイルパスは /proc/<pid>/ 配下を読んで取得します。

競合に注意

exec イベントが届いてから /proc/<pid>/ を読むまでの間に、 極短命なプロセスは消えていることがあります。コードは FileNotFoundError を捕捉して None を返すようにしていますが、その場合 cmdline が取れないため イベントは破棄します（cmdline なしでは検知に意味が無いため）。

制限と既知の挙動

  • fork のみのイベントは取れない: EXEC を購読しているため、fork(2) の後 exec(2) しないプロセス （長期常駐の bash と同名 binary など）は記録されません。 通常は親プロセスから fork → exec の流れなので問題になりません。
  • extreme short-lived プロセス: netlink から通知が届いた直後に /proc/<pid>/ が消えるほど 短命なプロセスは捨てます（数 ms オーダーの終了）。
  • コンテナ識別なし: cgroup 情報を見ていないため、Docker / containerd 内のプロセスも ホスト側の pid で混在する。コンテナ識別は未実装。
  • 権限: CAP_NET_ADMIN が必要。systemd ユニットでは AmbientCapabilities=CAP_NET_ADMIN + User=ursus で 最小権限化が可能です（現状は実装未着手）。

eBPF + LSM hook との比較

商用 EDR の主流は eBPF + LSM hook。 security_bprm_check や execve tracepoint に bpf プログラムを attach することで、cn_proc では取れない以下が可能になる。

  • cgroup / namespace / capability set など task_struct の任意フィールド取得
  • カーネルから user space への効率的な ring buffer 転送
  • LSM hook を使えば exec の 拒否もできる (block_process 系アクションの実装基盤)

URSUS は cn_proc + Python struct による実装を採用しており、bcc / libbpf 等の追加依存を持たない。トレードオフは上記 3 点の機能を諦めること。 詳しくは glossary: eBPF。
